# Abbreviations & Glossary

Every scary word in this course decoded in one sentence.

Core Git Terms

VCS
Version Control SystemSoftware that tracks every change to files. Git is a VCS — the most popular one.
REPO
RepositoryA folder Git is tracking. Every project = one repo. Contains all files + full change history.
COMMIT
Save SnapshotA permanent saved checkpoint. Like hitting "Save Game" — you can always come back to any save.
BRANCH
Parallel TimelineAn independent line of development. You can have many; main is the official one.
MERGE
Combine BranchesTakes changes from one branch and applies them to another. Can cause conflicts.
HEAD
Current Position"You are here" pointer. HEAD = whatever branch/commit you're currently on.
ORIGIN
Default Remote NameNickname for your remote repo on GitHub. "origin" = the GitHub version of your project.
REMOTE
Online VersionA repo hosted elsewhere (GitHub, GitLab). Local = your computer; Remote = the cloud.
STAGING
Pre-Commit AreaFiles added with git add but not committed yet. Like a "to be saved" tray.
INDEX
Alias for StagingGit internally calls the staging area the "index." Same thing, different name.
STASH
Temporary HideSaves uncommitted work in a hidden drawer so you can switch branches without losing changes.
REBASE
Rewrite HistoryMoves your branch's starting point. Cleaner than merge, but rewrites commit history — use carefully.
REVERT
Undo a CommitCreates a NEW commit that cancels out an old one. Safe undo — history stays intact.
RESET
Undo ChangesMoves HEAD back. --soft keeps changes staged, --mixed unstages, --hard deletes them.
DIFF
Show DifferencesShows exactly what lines changed between two states. Green = added, Red = removed.
LOG
History Listgit log shows all commits: who, when, message. Your project's diary.

GitHub-Specific Terms

PR
Pull Request"Please review and merge my branch." Not actually pulling — it's a request to merge code.
GH PAGES
GitHub PagesFree static website hosting from a GitHub repo. Your repo becomes a live website instantly.
SSH
Secure ShellConnect to GitHub without a password every time. Uses a key pair (lock + private key on your PC).
2FA
Two-Factor AuthLogin = password + code from your phone. Much safer. GitHub strongly recommends it.
gh CLI
GitHub CLI ToolOfficial command-line tool to manage GitHub (PRs, issues, repos) directly from your terminal.
API
Application Programming InterfaceA way for programs to talk to GitHub automatically. Used for automation and integrations.
LGTM
Looks Good To MeCommon PR review comment meaning "I approve this." Developer shorthand for approval.
CI/CD
Continuous Integration/DeploymentAutomated testing and deployment when you push code. GitHub Actions handles this.
UPSTREAM
Original Repo (in forks)When you fork a repo, the original is your "upstream." Pull from upstream to stay current.
PAT
Personal Access TokenA generated password substitute for HTTPS authentication. Use instead of your real password.
BLOB
Binary Large ObjectGit stores each file version internally as a blob. You'll see this in GitHub URLs.
LFS
Large File StorageGit extension for tracking large binary files (images, videos). Keeps your repo lean.

# Lab Reference Guide — All 10 Practicals

Step-by-step commands for every lab. Copy, run, understand. Each lab builds on the last.

LAB 1

Create local repo, commit files, connect to GitHub

Goal: Your files → Git tracking → GitHub. The complete journey from zero to cloud.
1
Set your identity (only needed once).
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
# Verify: git config --list
2
Create a project folder and initialize Git.
mkdir my-first-project
cd my-first-project
git init
# A hidden .git folder is created — Git now watches this folder.
3
Create a file and make your first commit.
echo "# My First Project" > README.md
git add . # stage everything
git status # see what's staged
git commit -m "Initial commit: add README"
4
Create a new empty repo on GitHub.com, then connect and push:
git remote add origin https://github.com/YourUsername/my-first-project.git
git branch -M main
git push -u origin main
# -u sets default so future pushes just need: git push
After the first push with -u, type just git push and git pull — no extra arguments needed.
LAB 2

Branches — Create, Switch, Edit, Merge

Goal: Work in a branch, keep main safe, and merge when done.
1
Create and switch to a new branch.
git checkout -b feature/add-about-page
# Modern alternative: git switch -c feature/add-about-page
git branch # asterisk (*) shows current branch
2
Make changes and commit on this branch.
echo "About page content" > about.md
git add about.md
git commit -m "Add about page"
3
Switch back to main and merge.
git checkout main
git merge feature/add-about-page
git branch -d feature/add-about-page # clean up
git push origin main
Never commit directly to main in a team project. Always work on a feature branch first!
LAB 3

Exploring the GitHub Interface

Goal: Navigate GitHub's web UI — understand what every button and page does.
1
Go to your repo → click the clock icon (N commits) to see history. Compare with git log --oneline in terminal.
2
Click any file and try the three buttons: Raw (plain text), Blame (who wrote each line — same as git blame), History (commits touching this file — same as git log -- filename).
3
Watch = get notified of activity. Star ⭐ = bookmark / show appreciation. Fork 🍴 = make your own GitHub copy.
4
Edit the repo description: click the ⚙️ gear icon next to "About" on your repo homepage.
Press T on any GitHub repo to instantly search files. Press ? anywhere for all keyboard shortcuts.
LAB 4

Issues — Track Tasks, Bugs & Milestones

Goal: Use GitHub Issues like a to-do + bug tracker for your project.
1
Create an issue: Issues → New Issue. Write a title, description, and assign it to yourself.
2
Create custom labels: Issues → Labels → New Label. Try: bug (red), enhancement (blue), good first issue (purple).
3
Create a Milestone: Issues → Milestones → New Milestone. Name it "Sprint 1", set a due date. Add issues to it. GitHub shows a progress bar.
4
Auto-close an issue via commit message:
git commit -m "Fix broken nav — Closes #1"
# Keywords: Closes, Fixes, Resolves (+ #issue-number)
git push
Use Fixes #3 in your PR description on GitHub to auto-close Issue #3 when the PR is merged.

GitHub Organizations & Teams

Fork a Repo, Create a PR & Peer Review

Goal: The full open-source contribution workflow — Fork → Branch → Change → PR → Review → Merge.
1
Fork a classmate's repo: click Fork on their repo page. GitHub creates YourUsername/their-repo.

Clone your fork and add upstream:

3
Create a branch, change, push to YOUR fork, then open a PR:
git checkout -b fix/typo-readme
# edit the file...
git add . && git commit -m "Fix typo in README"
git push origin fix/typo-readme
# GitHub: click "Compare & pull request" → submit PR

Peer review: click Files changed tab, click the + on any line for inline comments. Use Review changes → Approve or Request changes.

LAB 9

Repository Settings — Control Your Repo

Goal: Learn all key repository settings for maintenance, security, and collaboration.
1
Change default branch: Settings → General → Default branch → rename to "main".
2
Enable/Disable Wiki & Issues: Settings → General → Features. Toggle on/off as needed.
3
Add a Collaborator: Settings → Collaborators → Add people → type their username. They get an email invite.
4
Transfer repo ownership: Settings → Danger Zone → Transfer. Type the repo name to confirm. Can transfer to another user or org.
5
Delete a repo: Settings → Danger Zone → Delete. Type the full repo name. This is permanent and irreversible.
Danger Zone actions are irreversible. GitHub forces you to type the repo name as confirmation — do it slowly and deliberately.
LAB 10

Profile, SSH Keys, 2FA & Notifications

Goal: Secure your GitHub account and set up passwordless SSH authentication.
1
Update Profile: github.com → Your profile → Edit profile. Add bio, college, and a profile picture.
2
Set up SSH Key:
# Generate a key pair:
ssh-keygen -t ed25519 -C "you@example.com"
# Copy your PUBLIC key:
cat ~/.ssh/id_ed25519.pub
# Add to GitHub: Settings → SSH Keys → New SSH Key → Paste → Save
# Test:
ssh -T git@github.com
# Should say: Hi YourName! You've successfully authenticated.
3
Enable 2FA: Settings → Password and Authentication → Enable 2FA. Use Google Authenticator or Authy. Save your recovery codes somewhere safe offline!
4
Notifications: Settings → Notifications. Keep "Participating" on; turn off "All Activity" to avoid email spam.
5
Switch existing repos from HTTPS to SSH:
git remote set-url origin git@github.com:YourUsername/repo.git
git remote -v # verify
NEVER share ~/.ssh/id_ed25519 (private key). Only the .pub file goes to GitHub. The private key stays on your computer forever.

# Troubleshooting — When Things Break

The most common errors you will hit — every developer faces these. Now you'll know exactly what to do.

🔥 The Big One: Merge Conflicts

💥

MERGE CONFLICT — "Automatic merge failed; fix conflicts and then commit"

What happened: Two people (or two branches) edited the same line. Git can't decide which version to keep — it needs YOU to decide.
# Git marks the conflict in the file like this:
<<<<<<< HEAD
Hello from the main branch
=======
Hello from the feature branch
>>>>>>> feature/new-greeting

# <<<<<<< HEAD = YOUR current version
# ======= = the divider
# >>>>>>> = the incoming version
How to fix: Open the file, delete the conflict markers (<<<, ===, >>>), edit to the version you want, then:
git add conflicted-file.md
git commit -m "Resolve merge conflict"
VS Code highlights conflicts with colour-coded "Accept Current / Accept Incoming / Accept Both" buttons. Much easier than editing manually.
❌

REJECTED PUSH — "rejected … Updates were rejected because the remote contains work you do not have"

What happened: Someone else pushed to the same branch before you. Your local version is behind GitHub's version.
Fix:
git pull origin main # or: git pull --rebase origin main
# If conflict arises, resolve it, then:
git push origin main
NEVER do git push --force on a shared branch. It overwrites everyone else's work permanently.
🔒

SSH — "Permission denied (publickey)"

What happened: SSH isn't set up, the key wasn't added to GitHub, or the agent doesn't have the key loaded. On Windows, you may need to run ssh-add from Git Bash as administrator.
Fix:
ssh-add -l # check if key is loaded
ssh-add ~/.ssh/id_ed25519 # load it if not
ssh -T git@github.com # test connection
ssh -vT git@github.com # verbose debug if still failing
😱

DETACHED HEAD — "You are in 'detached HEAD' state"

What happened: You checked out a specific commit instead of a branch. Commits you make here can be lost.
Fix:
# Option 1: Go back to main (discard commits from detached state):
git checkout main
# Option 2: Save your work by creating a branch here:
git checkout -b saved-from-detached-head
😬

WRONG COMMIT MESSAGE — Already committed (not yet pushed)

What happened: You typed a bad commit message and want to fix it before pushing.
Fix (only if not yet pushed):
git commit --amend -m "New, corrected commit message"
Once pushed to a shared branch, leave it — amending requires a force push which rewrites shared history.
🗑️

ACCIDENTAL DELETION / WRONG CHANGES — Need to undo

What happened: You made changes or deleted something you shouldn't have.
Fix — choose based on situation:
# Discard ALL uncommitted changes (irreversible!):
git restore .
# Restore ONE specific file:
git restore path/to/file.txt
# Unstage a file (keep changes):
git restore --staged file.txt
# Undo last commit, keep changes staged:
git reset --soft HEAD~1
# Undo last commit AND delete all changes (nuclear):
git reset --hard HEAD~1
🌐

GITHUB PAGES NOT SHOWING — Site is blank or 404

What happened: Pages isn't configured correctly or hasn't deployed yet.
Checklist:
  • Wait 2–3 minutes after enabling Pages — deployment takes time
  • File must be named exactly index.html (lowercase, at repo root)
  • Settings → Pages → ensure source branch is set correctly
  • Check the Actions tab — a failed build shows as a red X
  • Hard-refresh your browser: Ctrl+Shift+R
🔀

STALE FORK — My fork is behind the original repo

What happened: The original repo got new commits after you forked. Your fork is outdated.
Fix:
git fetch upstream
git checkout main
git merge upstream/main
git push origin main
# Or: on GitHub.com, click the "Sync fork" button (much easier!)
🔐

AUTHENTICATION FAILED — "Support for password authentication was removed"

What happened: GitHub no longer accepts plain passwords over HTTPS since August 2021. Use a PAT or SSH.
Fix — use SSH (see Lab 10) OR create a Personal Access Token:
  • Settings → Developer settings → Personal access tokens → Tokens (classic)
  • Generate token → select repo scope → copy token
  • Use this token as your password when Git asks

1

The Foundation

Git ≠ GitHub — Know the difference

🧠

Git

The software on your computer that tracks file changes. Lives in your terminal. Works 100% offline. Like a time machine for your code.

☁️

GitHub

The website (github.com) that hosts your Git repos online. Adds team features, issue tracking, and a web UI on top of Git.

Analogy: Git is like Microsoft Word (the software). GitHub is like Google Drive (the cloud). You write and save in Word — you share and collaborate via Drive.
Core Concept

What is a Repository (Repo)?

A repo is a project folder that Git is watching. It stores all your files + the entire history of every change ever made.
📁

Local Repo

The copy on your computer. You edit files here.

☁️

Remote Repo

The copy on GitHub. Your team syncs through this.

Core Concept

Branches — Parallel Timelines

main is the "official" version. Branches let you experiment without breaking it.
1
You're on main — the safe, working version
2
Create feature/dark-mode — a parallel universe
3
Build, break, experiment — main is untouched
4

# make changes, commit...
git checkout main
git merge feature/dark-mode
Analogy: main is the published book. A branch is your draft on a notepad. When the draft is good, you add it to the book.

Add repos to the team with a permission level.

Core Concept

What is a Pull Request?

A PR is a formal request to merge your branch into another. It's asking: "Can someone review my changes before we merge?"
Your Branch
→
Open PR
→
Review
→
Approve
→
Merge
Analogy: A PR is like submitting an assignment. The teacher (reviewer) checks your work, writes feedback, and either approves or requests corrections.
GitHub calls them "Pull Requests". GitLab calls them "Merge Requests" (MR). Same thing — different names.
Collaboration Models

Two Ways to Collaborate

🏠

Shared Repository Model

Everyone has push access to the same repo. Create a branch, commit, open a PR to main.


Best for: Team projects, class assignments, company teams.

🍴

Fork & Pull Model

Fork someone else's repo, make changes in your fork, open a PR to the original.


Best for: Open-source contributions — you don't have push access to the original.

Workflow

Creating a Pull Request — Step by Step

1
Push your branch: git push origin feature/my-feature
2
GitHub shows a yellow banner: "Compare & pull request" — click it.
3
Set base branch (destination — usually main) and compare branch (your feature branch).
4
Write a clear title & description. Add Closes #5 to link to an issue.
5
Assign a reviewer, add labels, link a milestone. Click Create Pull Request.
Don't create a PR from main → main. Always branch first, then PR. PRs compare two different branches.
Code Review

The Review Process

As a Reviewer:

👁️
Files changed tab — see what lines were added (green) or removed (red).
💬
Click the + on any line to leave an inline comment.
📝
Review Changes: Comment, Approve, or Request Changes.

As the Author:

📬
Respond to comments. Push new commits — PR updates automatically.
✅
Once approved, click Merge Pull Request.
🗑️
Delete the feature branch after merging.
Merging & Undoing

Three Merge Types + Reverting

🔗

Merge Commit

Default. Creates a merge commit showing branches joined. Preserves full history. Safe for most cases.

📦

Squash & Merge

Combines all branch commits into ONE on main. Keeps history clean. Good for messy "WIP" commit histories.

🔀

Rebase & Merge

Replays commits on top of main — no merge commit. Cleanest history but rewrites commits. Advanced.

Merged a bad PR? Click Revert on the merged PR — GitHub creates a new PR that undoes it. History stays intact.

Security

SSH Keys — Passwordless Authentication

SSH uses a key pair: private key on your PC (like a physical key) and public key on GitHub (the lock). Once set up, you never type a password again.
# Step 1 — Generate key pair:
ssh-keygen -t ed25519 -C "you@email.com"

# Step 2 — Copy YOUR PUBLIC key:
cat ~/.ssh/id_ed25519.pub

# Step 3 — GitHub: Settings → SSH Keys → New SSH Key → Paste

# Step 4 — Test:
ssh -T git@github.com
# Should say: Hi YourName! You've successfully authenticated.
NEVER share ~/.ssh/id_ed25519 (the private key). Only the .pub file goes to GitHub. Private key stays on your PC forever.
Security

Two-Factor Authentication (2FA)

2FA = password + code from your phone. Even if someone steals your password, they can't log in without your phone.
1
Settings → Password & Authentication → Enable 2FA
2
Download an app: Google Authenticator, Authy, or Microsoft Authenticator
3
Scan the QR code shown on GitHub
4
SAVE recovery codes offline — losing your phone without these = locked out permanently
🔐

How 2FA Login Works

1. Enter username + password.


2. GitHub asks for 6-digit code.


3. Open authenticator → copy rotating code → enter it.


4. Done. Code changes every 30 seconds.
